Why validation waits for spring
Through autumn/winter, black-type form is near-universally 0-0-0 — the program is maidens and midweek grades where almost nothing in the field has stakes form. Validating on winter data would give a tiny non-zero sample and a false read (the exact thin-sample trap that produced false leads earlier). So v37.2 banks the data now and defers the validation to spring, when Group/Listed racing populates the fields and the sample carries real class spread.
Why it's likely a metro-race signal
Black-type form travels across venue and distance — the cross-venue class comparison the model currently lacks (it leans on coarse race-type labels). But it's blank for exactly the country/provincial maidens the model bets most, so it will carry information mainly where a field has genuine class spread: metro and stakes races. Expectations tempered accordingly — a refinement where it applies, not a broad signal.
How it's built (data plumbing)
Persisted to eight nullable bt_* columns on the runners table — blank for historical rows and any day the data is absent, so nothing breaks. A v_blacktype_coverage view tracks non-zero black-type runners per week, so we can watch the sample build toward a spring validation and know when it's finally large enough to test. In v37.3 this collection folds into the single combined course file and the normal runner insert (the interim separate file and its parser are retired).
Roadmap position
v37.2 is deliberately separated from the v38 fitted-improvement batch: v38 changes bet selection and needs the full betting-replay regression; v37.2 touches no score or gate and only writes new columns, so it ships independently and carries a lighter check. Spring: validate black-type at runner level (win rate by highest-class band vs market-implied), and score it only if it beats its market price — shadow-logged first, like every signal before it.
