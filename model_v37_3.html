The consolidation
Input went from three files per course to one. Decisions are byte-identical — this changes how the model is fed, not what it decides.
Before — three files

Main T-file

Form/scoring (positional, 16 cols)

Dup "(1)" file

AES/speed + extended stats — collision-named, browser-suffix dependent

Black-type file

G1/G2/G3/LR — sniffed by content to survive the name clash
After — one file

One combined XLS per course

26 columns, header-keyed (order-independent). Form + speed + black-type in one sheet.

Per-race CSVs

Unchanged.

Retired

The dup loader, the black-type sniffing parser, and SQL5 all gone — net simpler.
The 26 columns
Tab · Horse · Form L5 · BP · P · Jockey · JRat · Trainer · TRat · Wgt · FU · 2U · 3U · DC · LS Det · PP · AES · AFS · RLW · DLW · WTC · DLR · G1 · G2 · G3 · LR

Read by header name, not position, so columns can be reordered freely — only the labels must match. This is more robust than the old positional main-file parse, which shifted if a column moved. API was dropped (v35 found it flat/redundant) to keep the sheet lean.
Equivalence — the test that mattered
Consolidating inputs must not change a single bet. It didn't.
Method
The same source data was fed through both the old three-file pipeline and the new single-file one (split from one combined file to eliminate any data difference), and every runner's decision compared.

A useful confound
A naive first comparison showed 35 "differences" — all traced to the source data being refreshed a day apart (prices, barriers, jockey swaps). Controlling for it drove the difference to zero, and confirmed the parser is faithful.
Deployment note
Live betting behaviour is v37.1 — v37.3 is an input/plumbing change only. Black-type now persists through the normal runner insert (the bt_* columns), so a correct deployment shows any bt_runs > 0 runner in v_blacktype_coverage for the day. If those come back null, the run used the old file format. The daily pull moves to tag v37.3; DAILY_RUN.md documents the single-file workflow.
